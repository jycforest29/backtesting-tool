Backtesting & Trading System — Technical Architecture

한화증권 계좌 기반 퀀트 실매매 + 백테스트 인프라. 본 문서는 구현의 "무엇" 이 아닌 "왜" 를 담는다.
요약. 실매매 장애대응 중심의 아키텍처. 10개의 회복 패턴이 겹겹이 깔려있고, 142개 테스트가 그 계약을 고정한다. 핵심 불변식: 주문은 마지막까지 살린다, event 는 무조건 Kafka 로 간다, audit log 는 변조 불가능하다, 어떤 downstream 장애도 다른 downstream 에 전파되지 않는다.

1. Design Principles

시니어의 도구상자가 아니라, 결정 기준이다. 새 코드 한 줄도 아래 기준을 통과하지 못하면 기각.

  1. 장애는 섞이지 않는다 (isolation). KIS 가 느려져도 DART 호출은 영향 없고, 백테스트가 크래시해도 주문 API 는 산다. 구현: 별도 스레드 풀(bulkhead), 별도 서킷 브레이커, priority 기반 admission.
  2. 돈 관련 작업은 at-least-once. 주문 기록과 trade-events Kafka 발행은 같은 DB 트랜잭션이 아니다. Outbox 가 그 둘을 원자적으로 잇는다. 소비자는 event.id 기준 dedup 의무.
  3. 멱등성은 구조적으로 보장한다. 클라이언트의 X-Idempotency-Key 가 있으면 재시도 시 캐시 응답. 같은 키+다른 body 는 409. Stripe 패턴 호환.
  4. degradation 은 침묵하지 않는다. 모든 응답은 circuitState · degradedReason · replay 같은 메타를 들고 나간다. 클라이언트가 상태를 알 수 있어야 대응한다.
  5. 시간은 주입된다. Instant.now() 대신 Clock. 테스트·DST 에서 시간 압축 가능.
  6. 모든 불변식은 테스트로 고정. 142 개 중 80% 가 "이게 어겨지면 돈이 샌다" 수준의 invariant 검증.

2. Fault Tolerance Layers

요청 하나가 시스템을 통과할 때, 겹겹의 방어선이 있다.

HTTP request │ ▼ [1] DeadlineFilter ← X-Deadline-Ms 로 예산 설정, 504 처리 │ ▼ [2] AdmissionFilter ← Priority 기반 load shedding, 503 │ ▼ [3] Idempotency layer ← X-Idempotency-Key 재시도 흡수 │ ▼ [4] Business logic ← OrderExecutionService │ ├─→ KIS HTTP call → [5] Bulkhead pool (kisExecutor) │ → [6] Rate limiter (slid window) │ → [7] Circuit breaker │ → [8] Tracing span │ └─→ DB commit ← @Transactional └─→ Outbox row INSERT (같은 tx) ↓ [9] OutboxPublisher (scheduled, two-phase) └─→ Kafka at-least-once

3. Transactional Outbox + Idempotency

문제

주문 한 번이 KIS HTTP 성공 → DB 기록 → Kafka 이벤트 발행의 3단계를 거친다. 이 3개가 각자 tx 라면 dual-write inconsistency: DB 에는 주문이 있는데 Kafka 에는 이벤트가 없음 (다운스트림 분석 시스템 유실), 또는 반대.

구현의 정수

@Transactional
void persist(OrderRequest req, OrderResult result, String idempotencyKey) {
    orderRepo.save(record);        // 주문 영속화
    outbox.record(..., event);     // 같은 tx 에서 outbox row INSERT
}                                   // 커밋 — 둘 다 남거나, 둘 다 롤백

// 별도 워커 (독립 tx)
@Scheduled(fixedDelay=2s)
public void drainBatch() {
    for (event : repo.findUnsent(...)) {
        publishOne(event.id);       // Kafka 로 발행
    }
}

핵심 불변식

DB 에 커밋된 모든 OutboxEvent 는 유한 시간 내 반드시 Kafka 에 한 번 이상 발행된다.

Two-Phase publish (큰 개선)

초기 구현은 publishOne 자체가 @Transactional 이라 Kafka send().get() 대기 동안 DB 커넥션을 점유했다. 프로덕션(Postgres)이면 connection pool 고갈. 수정: 

public void publishOne(String id) {      // NO @Transactional
    var e = repo.findById(id).orElse(null);
    if (e == null || e.getSentAt() != null) return;
    try {
        kafka.send(...).get(timeout);     // 네트워크 I/O — 커넥션 점유 없이 대기
        markSent(id);                      // @Transactional(REQUIRES_NEW)
    } catch (Exception ex) {
        markRetry(id, ex);                 // 같은 패턴
    }
}
Trade-off. 이 분리로 crash window 가 확장된다 — Kafka send 성공 직후 markSent 전에 크래시 시 중복 발행. 수용 가능: at-least-once + consumer dedup 이 공식 계약이므로 소비자가 event.id 로 중복 제거.

Idempotency 프로토콜

Stripe 호환. 4가지 상태:

시나리오응답
신규 키실행 → X-Idempotent-Replay: false
같은 키 + 같은 body + COMPLETED캐시된 응답 → X-Idempotent-Replay: true
같은 키 + 다른 body409 Conflict (키 재사용 오류)
같은 키 + IN_FLIGHT409 InFlight (동시 처리 중)
같은 키 + FAILED409 — 새 키로 재시도

원자성은 DB UNIQUE(idempotency_key) 제약이 담당. 두 요청이 같은 키로 동시에 들어와도 INSERT 는 한쪽만 성공, 패배자는 DataIntegrityViolationException → 기존 row 조회 분기.

4. Deadline Propagation + Bulkhead

문제

두 가지 전형적 프로덕션 장애:
(a) KIS 가 20초 hang 하는데 프런트는 30초 타임아웃 → 프런트가 무감각.
(b) KIS 가 느려져 공용 HttpClient 풀이 포화 → 관련 없는 DART 호출도 대기.

해법 1 — Deadline

요청 진입 시 X-Deadline-Ms 또는 기본 30s 로 예산 설정. ThreadLocal RequestContext 에 deadline 저장. HTTP 호출마다: 

Duration timeout = Duration.ofMillis(
    RequestContext.effectiveTimeoutMs(DEFAULT_TIMEOUT_MS));
if (timeout.isZero()) throw new DeadlineExceededException(...);
// HttpClient.sendAsync(..., timeout)

해법 2 — Bulkhead

@Bean public ExecutorService kisExecutor() { return build("kis", props.getKis()); }
@Bean public ExecutorService dartExecutor() { ... }
@Bean public ExecutorService elwExecutor() { ... }

각 풀은 AbortPolicy — 포화 시 RejectedExecutionException 즉시 throw. CallerRuns 를 쓰지 않는 이유: 호출 스레드가 downstream 일을 대신 하면 격리가 깨진다.

핵심 불변식

KIS 풀 saturation 은 DART 풀 처리시간에 영향을 주지 않는다.

BulkheadIsolationTest — KIS 풀 2+1 슬롯 모두 점유한 상태에서 DART 풀 요청이 500ms 이내 완료 검증.

Async 컨텍스트 전파

public final class ContextPropagatingExecutor implements ExecutorService {
    private Runnable wrap(Runnable r) {
        RequestContext ctx = RequestContext.current();
        Map<String,String> mdc = MDC.getCopyOfContextMap();
        io.opentelemetry.context.Context otel = Context.current();
        return () -> {
            if (ctx != null) RequestContext.set(ctx);
            if (mdc != null) MDC.setContextMap(mdc);
            try (Scope ignored = otel.makeCurrent()) { r.run(); }
            finally { RequestContext.clear(); MDC.clear(); }
        };
    }
}

CompletableFuture.supplyAsync(..., bulkheadPool) 에서도 deadline + traceId + OTEL context 가 자식 스레드로 자동 전파. 이게 없으면 async 경계에서 trace 끊기고 deadline 잃고 로그 correlation 안 되는 — 전형적 senior-middle 함정.

5. Load Shedding + Admission Control

세 계층 priority

Priority용도거부 기준
CRITICAL주문, 청산, 포지션 조정절대 거부하지 않음 (시스템 살아있는 한)
STANDARD조회, 대시보드, 잔고hardLimit 초과 시만
BULK백테스트, preview, 팩터 스캔softLimit 초과 시 먼저 버림
public boolean tryAcquire(Priority p) {
    int cur = inFlight.get();
    if (cur >= hardLimit) {
        return p == Priority.CRITICAL && inFlight.incrementAndGet() > 0;
    }
    if (cur >= softLimit && p == Priority.BULK) return false;
    inFlight.incrementAndGet();
    return true;
}

URL 기반 자동 분류

if (path.startsWith("/api/trading/order"))  return Priority.CRITICAL;
if (path.startsWith("/api/quant/backtest")) return Priority.BULK;
if (path.startsWith("/api/elw/skew"))       return Priority.BULK;
return Priority.STANDARD;
과부하 상황에서 백테스트는 503 을 받지만 주문은 통과한다.

6. Circuit Breaker + Freshness Propagation

상태 기계

CLOSED  ──(5 failures / 60s)──▶ OPEN
  ▲                              │
  │                              │(30s cooldown)
  │                              ▼
  │       probe success    HALF_OPEN
  └──────────────────────       │
                                ▼(probe failure)
                              OPEN

동시성 불변식 (중요)

HALF_OPEN 상태에서 probe 실행 중, 다른 스레드의 호출은 람다를 실행하지 않는다.

초기 구현에서 놓친 것: state == OPEN 만 체크하면 HALF_OPEN 상태 중 도착하는 다른 요청들이 람다를 실행해 probe 의미가 사라짐. 동시성 테스트가 이 버그를 잡았다. 수정: 

public <T> Optional<T> protect(Callable<T> call) {
    while (true) {
        State s = state.get();
        if (s == State.CLOSED)   return runCall(call);
        if (s == State.HALF_OPEN) return Optional.empty();  // 다른 probe 진행 중
        // OPEN
        if (now - openedAt < COOLDOWN_MS) return Optional.empty();
        if (state.compareAndSet(OPEN, HALF_OPEN)) return runCall(call); // 나만 probe
        // CAS 패배 — loop 재시도
    }
}

Freshness — 침묵 금지

ElwIvRow 는 성공·실패·stale 행 모두 응답에 포함되고 ivStatus + solverIterations + asOf 를 달고 다닌다. 프런트는 CONVERGED 아닌 행을 회색 처리해 confidence 시각화. ElwSkewResponse.degradedReason 은 서비스 전체 degradation 을 투명화.

실패를 조용히 숨기지 않는다. 모든 응답은 자신의 정확도를 스스로 말한다.

7. Tamper-Evident Audit (Hash Chain)

문제

단순 append-only 테이블은 DBA 권한이나 insider 의 UPDATE 로 변조 가능. 국내 금감원 내부통제 기준 / MiFID II 감사 로그 요구사항은 tamper evidence 를 요구.

체인

row(1).prevHash = GENESIS
row(1).thisHash = SHA-256(1 | occurredAt | actor | action | payload | GENESIS)
row(2).prevHash = row(1).thisHash
row(2).thisHash = SHA-256(2 | ... | row(1).thisHash)
...

Canonical hashing (빅테크 디테일)

잘못된 구현: md.update(occurredAt.toString()). Instant.toString() 은 ISO 포맷이긴 하지만 nanosecond 유무가 상황에 따라 다름 (2024-01-01T00:00:00Z vs 2024-01-01T00:00:00.123456789Z) — 동일 의미, 다른 hash. 

md.update(longBytes(seq));
md.update(longBytes(occurredAt.getEpochSecond()));
md.update(intBytes(occurredAt.getNano()));
writeLengthPrefixed(md, actor);       // 4 bytes length + UTF-8 bytes
writeLengthPrefixed(md, action);
writeLengthPrefixed(md, payload);
writeLengthPrefixed(md, prevHash);

Length-prefix 는 필드 경계 불명확성 제거. actor="a", action="b" 와 actor="ab", action="" 가 같은 byte stream 을 만들지 않도록. 이게 domain separation — 암호학 원칙.

Verifier

for (AuditChainEntity row : batch) {
    if (row.getSeq() != lastSeq + 1)                  throw TamperDetected("seq gap");
    if (!row.getPrevHash().equals(expectedPrevHash))  throw TamperDetected("link broken");
    if (!service.recomputeHash(row).equals(row.getThisHash())) throw TamperDetected("payload altered");
}
체인 변조는 전수 검증(@Scheduled)으로 최대 24h 내 탐지된다.

Sealed hierarchy

    Instant occurredAt();
    String eventType();

    record Enabled(..., BigDecimal initialAllocation) implements StrategyDomainEvent {}
    record Rebalanced(..., Map<String,BigDecimal> targetWeights, String rationale) implements ... {}

Jackson @JsonTypeInfo + @JsonSubTypes 로 polymorphic JSON. 새 이벤트 type 추가 시 StrategyAggregateState.apply 의 instanceof 사슬에 분기 없으면 IllegalStateException — 컴파일러 대신 런타임이 빠짐 감지 (Java 17 호환).

OCC (Optimistic Concurrency)

public long append(String aggId, long expectedVersion, StrategyDomainEvent e) {
    long currentMax = repo.findMaxSeq(aggId).orElse(0L);
    if (currentMax != expectedVersion) throw new OptimisticConcurrency(...);
    // INSERT (aggId, seq=expectedVersion+1, ...)
    try { repo.saveAndFlush(entity); }
    catch (DataIntegrityViolationException dup) {
        throw new OptimisticConcurrency(...);   // UNIQUE(aggId,seq) 백스톱
    }
}

2중 방어 — 체크 선행 + DB UNIQUE 제약. 체크와 INSERT 사이 경쟁조건이 발생해도 제약이 잡는다.

Time-travel

public StrategyAggregateState projectAt(String aggId, Instant asOf) {
    StrategyAggregateState state = StrategyAggregateState.empty(aggId);
    List<StrategyEventEntity> events = eventStore.loadUntil(aggId, asOf);
    for (var e : events) state = state.apply(deserialize(e), e.getSequenceNumber());
    return state.withAsOf(asOf);
}

snapshot 은 seq 기준이라 시점 기준 재생에 쓰지 않음. 순수 이벤트 fold 만 사용 → 깔끔. GET /api/strategy-events/VAA/state?asOf=2024-03-15T14:00:00Z

어떤 과거 시점의 전략 상태도 event log 로부터 재구성 가능하다 (audit-grade).

9. Black-Scholes IV Solver

숫자 수학

// Newton-Raphson + bisection fallback
for (int i = 1; i <= MAX_ITER; i++) {
    double[] pv = priceAndVega(s, k, t, r, q, sigma, isCall);
    double diff = pv[0] - marketPrice;
    if (Math.abs(diff) < PRICE_TOL) return IvResult.ok(sigma, i);
    if (pv[1] < MIN_VEGA) return bisect(...);                 // vega≈0 → bisection
    double step = diff / pv[1];
    double next = sigma - step;
    if (!(next > MIN_SIGMA) || !(next < MAX_SIGMA)) return bisect(...);  // 오버슛
    if (Math.abs(step) < SIGMA_TOL) return IvResult.ok(next, i);
    sigma = next;
}

실패를 Status 로

수렴 실패를 예외로 던지면 대량 호출(ELW 체인 100건)에서 하나가 전체를 오염. 대신 IvResult(Double value, int iterations, Status status, String note) 로 반환: 

CONVERGED / MAX_ITER / ZERO_VEGA / INVALID_INPUT / NO_SOLUTION

Property-based testing

jqwik 으로 5 개 속성 × 1,200 무작위 샘플:

  • Round-trip: σ → price → σ 복원 (near-ATM, vol≥15%, t≥0.1년 regime)
  • 상태-값 정합: CONVERGED ↔ value≠null
  • 단조성: price ↑ ⇒ IV ↑
  • 바운드 밖 → NO_SOLUTION
  • 풋-콜 대칭: 같은 σ 의 call/put 역산 결과 동일

실제 가치: 첫 실행에서 deep ITM + near expiry + low vol 조합에서 실패. 이게 regime 한계 (vega ≈ 0 일 때 σ 역산 민감도 낮음) 를 데이터로 드러냄. 테스트가 버그가 아닌 구현의 적용 영역 을 자동 발견.

11. Test Philosophy

142 테스트. 3 단계 피라미드.

계층도구역할
1. Unit (대부분)JUnit 5 + Mockitopure logic — 전략 signal, idempotency state machine 등
2. Propertyjqwik (1,200 samples)수학 invariant — regime 한계 자동 발견
3. Integration (HTTP)WireMock실제 HTTP roundtrip — 파싱·헤더·에러 경로
4. DST (최상위)custom deterministic시간 압축 + 다중 seed → 수렴 증명

테스트가 발견한 실제 버그

  1. Circuit breaker HALF_OPEN race. 동시성 테스트가 probe 실행 중 다른 스레드가 람다 실행하는 버그 적발.
  2. Outbox tx 내 Kafka I/O. 리뷰 중 발견 → two-phase 로 분리.
  3. IV solver regime 한계. jqwik 이 deep ITM + near expiry 에서 precision 한계 드러냄 → regime 명시화.
  4. DST 수렴 시간. backoff {1,5,30,120,600,3600s} + 50% fault + 20분 window 조합이 수렴 실패 → 파라미터 범위 문서화.

12. Deterministic Simulation Testing

왜 DST 인가

WireMock = "한 번의 HTTP roundtrip 테스트". DST = "수천 개의 fault 시퀀스를 결정론적으로 반복". FoundationDB 가 했던 방식.

설계

long[] seeds = {1, 42, 2026, 7777, 99991};
double[] faultRates = {0.10, 0.30};

for (long seed : seeds) {
    for (double rate : faultRates) {
        MutableClock clock = new MutableClock(start);
        List<OutboxEventEntity> store = seedEvents(20, clock);

불변식

모든 committed outbox event 는 유한 시간 내 Kafka 에 at-least-once 도착한다.

DST 가 증명한 것

  • 10 seed × 2 rate × 20 event = 400 무작위 시나리오 모두 수렴
  • backoff schedule + MAX_TICKS 조합의 수렴 여부를 수학 대신 실제 코드 실행으로 검증
  • 버그 발견 시 seed 만으로 재현 가능 — production 이슈 디버깅에 직접 활용 가능

13. Known Limits & Future Work

의도적으로 안 한 것

  • SKIP LOCKED on outbox. 현재 단일 publisher 가정. 다중 인스턴스 확장 시 native query 로 교체.
  • Adaptive concurrency limit. 현재 정적 threshold (32/48). 진짜 adaptive 는 p99 latency gradient 기반 (Netflix concurrency-limits). 트래픽 패턴 안정된 후 도입.
  • Distributed tracing auto-instrumentation. 현재 manual span. OTEL Java agent 로 전환하면 Spring MVC·JPA·Kafka 자동 계측. 운영환경 배포 시.
  • Event store snapshot for time-travel. 현재 snapshot 은 seq 기반 (current 용). asOf 시점 snapshot 은 시간축 bucket 별로 구성해야 해서 복잡도 급증 — 실측 성능 문제 전까진 미룸.
  • Chaos engineering in production. FaultInjector 는 test only. Toxiproxy + prod-lite 환경 에서 chaos 유도는 초기 2-3개월 안정화 이후.

다음 라운드 후보 (staff → principal)

  • Saga 패턴 — 다중 KIS 계좌 간 rebalance 의 원자성
  • Read model / CDC — CQRS 로 조회 성능 분리
  • Progressive delivery — canary deploys + 자동 rollback (실매매라 매우 보수적으로)
  • Mutation testing (PIT) — 142 테스트가 충분한지 mutants 로 검증
  • Tracing-based SLO — error budget 자동 계산
